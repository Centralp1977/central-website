# デビルアドボケイト（修正内容の批判的レビュー）

コード修正・画面最適化を行うたびに、修正者自身とは別の視点で「本当にこれで良いのか」を疑う専門レビュアーとして動く。楽観的な自己承認を防ぐことが目的。

## いつ実行するか

- HTML/CSS/JS を変更した直後、ユーザーに「完了しました」と報告する **前に必ず** 実行する
- 依頼された修正・レイアウト調整・レスポンシブ対応など、画面に影響するあらゆる変更が対象
- 単純な文言修正のみで見た目に影響しない場合は簡略化してよいが、レイアウト・サイズ・配置に関わる変更は必ずフル実行する

## チェック項目

1. **スコープ逸脱の確認**（[[scope-discipline-website-edits]]）
   依頼された要素だけを変更したか。関連しそうで未依頼の箇所まで触っていないか diff を見直す。

2. **3ブレークポイントでの実機検証**（[[responsive-check]] [[central-image-responsive]]）
   モバイル(375px)・タブレット(768px)・デスクトップ(1280px)で実際に preview を開き、以下を疑ってかかる：
   - テキストや画像がカード・コンテナからはみ出していないか（`scrollWidth` vs `getBoundingClientRect().width` を比較して数値で確認する。目視だけで「大丈夫そう」と判断しない）
   - グリッド/フレックスの列が `1fr` のみで `minmax(0,1fr)` になっていない箇所がないか（意図しない内容依存の拡大でオーバーフローする典型パターン）
   - 日本語段落に `word-break:keep-all` があるカードでは、英数字混在や長いカタカナ語で `overflow-wrap:anywhere` 相当のフォールバックがあるか
   - 修正した要素だけでなく、同じコンポーネントを使う他のカード・セクションにも同種の問題が潜んでいないか横展開して確認する

3. **既存ルールとの整合性**
   - ナビ・モバイルメニュー・メタタグの同期が必要な変更なら同時に反映したか（[[sync-section-heading-nav]] [[sync-mobile-menu-with-main-content]]）
   - `feature/xxx` ブランチで作業しているか（master 直コミットの禁止）（[[feature-branch-workflow]]）

4. **証拠の提示**
   「動作確認しました」ではなく、実際に preview で測定した数値・スクリーンショットなど、検証根拠を示す。根拠なく完了報告しない。

## 進め方

1. 修正を実装する
2. 上記チェックリストで自分の修正を疑う（性善説で流さない）
3. 問題が見つかったら報告前に自分で直す
4. 問題がなければ、何を確認して問題なしと判断したか（具体的な数値・画面幅）を簡潔にユーザーへ報告する
